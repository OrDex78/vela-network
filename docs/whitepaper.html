1. Abstract

Vela Network is an independent Layer-1 blockchain designed and implemented entirely from scratch in Rust. It is not a fork of any existing chain. Every component — consensus, networking, transaction processing, state management, and cryptographic signing — has been purpose-built to understand and demonstrate the fundamental mechanics of a modern blockchain system.

The network uses HotStuff BFT for consensus, libp2p for peer-to-peer communication, Ed25519 for digital signatures, and Blake3 for hashing. It runs a live testnet with a block explorer, web wallet, and faucet.

2. Introduction

Most blockchain projects start by forking an existing codebase — Geth, Substrate, Cosmos SDK — and customizing configuration. While practical, this approach obscures the underlying mechanics and limits deep understanding of the system.

Vela takes the opposite approach: build everything from first principles. The goals are:

  • Educational transparency — every line of code is written to be understood, not inherited.
  • Minimal dependencies — rely only on well-audited cryptographic and networking libraries.
  • Production-grade architecture — despite being educational, the system uses real BFT consensus, persistent storage, and proper transaction validation.

Vela is written in Rust for its memory safety guarantees, zero-cost abstractions, and excellent async runtime support via Tokio.

3. Consensus — HotStuff BFT

Vela implements the HotStuff Byzantine Fault Tolerant consensus protocol, originally published by Yin et al. (2019). HotStuff provides several advantages over classical BFT protocols like PBFT:

3.1 Why HotStuff?

  • Linear message complexity — each phase requires only O(n) messages, compared to O(n²) in PBFT.
  • Pipelined rounds — proposals, votes, and commits are overlapped across rounds for higher throughput.
  • Deterministic finality — once a block is committed, it is final. No probabilistic confirmation.
  • Leader rotation — a round-robin leader schedule ensures fairness among validators.

3.2 Three-Phase Commit

HotStuff Phases
Prepare → Leader proposes a block, replicas vote
Pre-Commit → Leader aggregates votes into a Quorum Certificate (QC)
Commit → Block is finalized with 2f+1 matching votes

3.3 Fault Tolerance

With n = 3f + 1 validators, the protocol tolerates up to f Byzantine (malicious or crashed) nodes. Vela's testnet runs 3 validators (f = 0 for liveness, demonstrating the protocol structure). The architecture supports scaling to any validator set size.

4. Network Layer

Vela's peer-to-peer networking is built on libp2p, the modular networking stack used by Protocol Labs, Ethereum 2.0, and Polkadot.

4.1 Transport

Nodes communicate over TCP with Noise protocol encryption and Yamux stream multiplexing. Every connection is authenticated and encrypted by default.

4.2 Message Propagation — Gossipsub

Blocks, transactions, and consensus messages are propagated using Gossipsub, a pubsub protocol optimized for blockchain workloads. Messages are published to three topic channels:

  • vela-blocks — new blocks and chain sync messages
  • vela-transactions — new transactions from wallets
  • vela-consensus — votes and proposals between validators

4.3 Peer Discovery

Local peers are discovered automatically via mDNS (multicast DNS). For remote peers, bootstrap addresses can be provided via CLI arguments or hardcoded in the binary. On startup, nodes send a SyncRequest to catch up on any blocks they missed while offline.

5. Cryptography

5.1 Digital Signatures — Ed25519

All transactions and consensus votes are signed using Ed25519 (RFC 8032), an elliptic curve signature scheme offering:

  • Fast verification — significantly faster than ECDSA (secp256k1)
  • Small keys — 32-byte private keys, 32-byte public keys, 64-byte signatures
  • Deterministic signatures — no random nonce needed, eliminating a class of implementation bugs

Addresses in Vela are the raw 32-byte Ed25519 public key, prefixed with vela: for human readability.

5.2 Hashing — Blake3

Vela uses Blake3 for all hashing operations: block hashes, transaction hashes, Merkle roots, and signing payloads. Blake3 is a cryptographic hash function that is:

  • Extremely fast — significantly outperforms SHA-256 and SHA-3
  • Parallelizable — designed for SIMD and multi-threaded hashing
  • Secure — based on the BLAKE2 and ChaCha families with a wide security margin

6. Tokenomics

Token Parameters
Name: VELA
Total Supply: 1,000,000,000 (1 billion)
Smallest Unit: nVELA (1 VELA = 1,000,000,000 nVELA)
Transaction Fee: Minimum 1 nVELA per transaction

6.1 Genesis Distribution

At genesis, the entire supply is allocated to a faucet address. This is a testnet design choice — the faucet allows any user to request tokens for testing. In a mainnet scenario, distribution would follow a more structured allocation.

6.2 Faucet

The testnet faucet distributes 10,000 VELA per request with a 24-hour cooldown per address. Faucet transactions are signed by the faucet's Ed25519 key and processed through the same transaction pipeline as user transactions.

6.3 Future: Validator Rewards

The architecture is designed to support validator staking and block rewards in future iterations. Leaders who successfully propose blocks will earn transaction fees collected from the included transactions.

Basic programmable transactions and on-chain logic execution
Phase 6 — Mainnet
Production deployment with expanded validator set, staking, and governance

8. Conclusion

Vela Network demonstrates that a fully functional Layer-1 blockchain can be built from scratch with modern tools. By choosing Rust, HotStuff BFT, libp2p, Ed25519, and Blake3, the project achieves a balance of performance, security, and clarity.

Every component is open source and designed to be read, studied, and extended. Vela is not just a blockchain — it is a reference implementation for anyone who wants to understand how blockchains actually work under the hood.

The full source code is available at github.com/OrDex78/vela-network.
